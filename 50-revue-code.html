✅ 1. Checklist de revue de code ERP — Back & Front

Cette checklist est le document de référence pour toute revue de code, audit de module ou validation de PR sur PetSuite.

🧭 1. Compréhension globale

☐Le code est compréhensible sans explication orale
☐Les noms (variables, fonctions, fichiers) sont clairs et explicites
☐La responsabilité du fichier est unique et identifiable
☐La logique métier est facile à suivre

🥳 2. Architecture & structuration

☐Le code respecte l'architecture définie (controller / service / repository)
☐Front : UI vs logique métier bien séparés
☐Back : API vs métier vs accès DB bien séparés
☐Aucun fichier "fourre-tout"
☐L'arborescence est logique et homogène

🔄 3. Factorisation & réutilisation

☐Aucun code dupliqué évident
☐Les logiques répétées sont mutualisées (fonctions, services, hooks)
☐Les composants UI sont réutilisables
☐Pas de copier/coller inutile

🧱 4. Composants Front-end

☐Les composants sont réutilisables et découplés
☐Pas de composants similaires en double
☐Les props sont claires et bien définies
☐Aucun composant ne contient de logique métier complexe

⚙️ 5. Logique métier

☐La logique métier est centralisée (services)
☐Aucune règle métier cachée dans les contrôleurs ou composants UI
☐Les règles métier sont explicites
☐Les cas particuliers sont gérés

🗁️ 6. Accès aux données / base de données

☐Les accès DB sont centralisés (repository / service)
☐Pas de requêtes dupliquées
☐Les requêtes sont optimisées
☐Les données sont manipulées proprement

🔄 7. Gestion des correctifs et évolutions

☐Les correctifs ne sont pas des "patchs" isolés
☐Les évolutions sont intégrées dans l'architecture existante
☐Le code ancien n'est pas contorné mais amélioré
☐Aucun "hack temporaire" laissé en production

🧾 8. Documentation & commentaires

☐Chaque fichier contient un commentaire d'en-tête
☐Les fonctions métier importantes sont documentées
☐Les règles métier sont expliquées
☐Les zones complexes ou sensibles sont commentées
☐Pas de commentaires inutiles ou obsolètes

🚨 9. Gestion des erreurs

☐Les erreurs sont gérées proprement
☐Pas de try/catch vides
☐Messages d'erreur compréhensibles
☐Pas d'erreurs silencieuses — logs présents si nécessaire

🔐 10. Sécurité (important ERP)

☐Validation des entrées utilisateur
☐Pas d'injection possible (SQL, XSS…)
☐Gestion des droits utilisateurs respectée
☐Aucune donnée sensible exposée

⚡ 11. Performance

☐Pas de requêtes inutiles
☐Pas de calculs répétés inutilement
☐Chargement optimisé côté front
☐Pas de boucle ou traitement lourd non maîtrisé

🧪 12. Tests (si présents)

☐Les fonctions critiques sont testées
☐Les tests sont lisibles et pertinents
☐Les cas métier principaux sont couverts

🎯 13. Qualité globale

☐Le code est lisible et propre
☐Pas de code mort ou inutile
☐Pas de debug laissé (console.log, etc.)
☐Respect des conventions d'équipe

🔍 2. Méthodologie approfondie

5 axes de la revue

AxeObjectifActions concrètes
1. UniformisationConvention de nommage homogèneVariables, fonctions, composants, patterns
2. FactorisationSupprimer les duplicationsFonctions utilitaires, services, composants réutilisables
3. StructurationOrganisation claireFront: pages/composants/services/hooks — Back: controllers/services/repositories
4. RéintégrationSupprimer les patchs bricolésRepenser les évolutions comme des fonctionnalités natives
5. DocumentationCommentaires utilesPourquoi (important) · Règles métier · Points sensibles

8 étapes de la méthodologie

  1. Étape 1 — Cartographie : Lister modules, composants, fonctions dupliquées. Identifier zones critiques et instables. Comprendre AVANT d'optimiser.
  2. Étape 2 — Détection duplications : Fonctions copiées, requêtes SQL similaires, composants UI identiques → créer une version unique.
  3. Étape 3 — Refactorisation progressive : ⚠ Ne jamais tout refaire d'un coup. Choisir un module → Nettoyer → Factoriser → Tester → Passer au suivant.
  4. Étape 4 — Couches claires : Back: Controller → Service → Repository. Front: Pages → Components → Services → Hooks/utils.
  5. Étape 5 — Documentation : En-tête de fichier + fonctions métier + points techniques sensibles.
  6. Étape 6 — Tests : Au moins sur les fonctions métier critiques.
  7. Étape 7 — Revue finale : Appliquer la checklist 13 points + scoring.
  8. Étape 8 — Mise à jour documentation : Mettre à jour ce document avec les résultats.

Règles de factorisation concrètes

RègleAction
🔶 Une logique répétée 2 fois→ À factoriser obligatoirement
🔶 Une logique métier dans un composant UI→ Jamais — extraire dans un service
🔶 Un accès DB dans un controller→ Jamais directement — passer par un service
🔶 Un composant copié→ Devient un composant générique avec props
🔶 Un if(role === 'admin') répété→ Extraire en fonction isAdmin(user)

Pièges à éviter

  • ❌ Tout refactorer d'un coup → casse garantie
  • ❌ Sur-abstraction → code illisible
  • ❌ Trop commenter → bruit inutile
  • ❌ Mélanger refacto + nouvelles features

Comment le code est créé

Stack technique

  • Frontend : Next.js 15 App Router, TypeScript, React, Tailwind CSS

Pattern API obligatoire

app.use('/ma-route', async (req: any, res: any) => {
  // Body parsing manuel obligatoire pour POST/PATCH
  const b: any = await new Promise<any>((resolve) => {
    if (req.body && Object.keys(req.body).length > 0) { resolve(req.body); return; }
    let data = '';
    req.on('data', (chunk: any) => { data += chunk; });
    req.on('end', () => { try { resolve(JSON.parse(data)); } catch { resolve({}); } });
  });
  const db = await mysql.createConnection({
    host: process.env.DB_HOST, port: Number(process.env.DB_PORT),
    user: process.env.DB_USER, password: process.env.DB_PASSWORD,
    database: process.env.DB_NAME
  });
  try {
    const [rows] = await db.execute("SELECT ...", [params]);
    res.json(rows);
  } catch(e: any) { res.status(500).json({ error: e.message }); }
  finally { await db.end(); }
});

🤖 6. Module IA PetSuite — Conception & Roadmap

Intégration de l'IA Claude directement dans PetSuite, accessible uniquement via les droits SuperAdmin pour la configuration, et disponible pour tous les utilisateurs autorisés.

Capacités sans modifier les écrans ni la BDD

CatégorieExemples de commandes
Assistant conversationnel"Crée un devis pour Dupont Jean, chien Jazz, consultation 2h, 120€, remise 10%"
Génération documents"Prépare le devis en PDF pour validation et envoi par mail"
Analyse données"Quelles sont mes factures impayées de plus de 30 jours ?"
Alertes proactives"3 clients sans séance depuis 60 jours" — "Stock X sous le seuil"
Rédaction"Rédige le compte-rendu de la séance de ce matin"
Rapport EC"Génère le rapport comportemental de Jazz à partir de mes notes"

Architecture technique

Composant : <AIAssistant />
- Bouton flottant persistent (bas droite)
- Panel lateral coulissant
- Acces aux routes API existantes (/api/...)
- Appel Anthropic API (claude-sonnet-4-20250514)
- Rendu resultats en contexte (modal PDF, pre-remplissage)
- Historique conversation en session
- Logs interactions en BDD (table ia_log)

Accès SuperAdmin uniquement

  • Configuration clé API Anthropic dans param_config
  • Activation/désactivation par utilisateur
  • Logs des interactions IA
  • Paramètres du prompt système (ton, langue, règles métier)
